6 # Projects
7 
8 The pencil2d project is organized into 3 sub-projects:
9 
10  - `app`: holding everything related to GUI, e.g, tool panel, colour wheel and Timeline etc.
11  - `core_lib`: the engine of Pencil2D, mostly about animation, drawing and tool manipulations.
12  - `tests`: a collection of unit tests.
13 
14 You can see these 3 subprojects in QtCreator when you open the Pencil2D project and find identical folder names in the repository.
15 
16 # Where to start with?
17 
18 At the time of writing, there are nearly 200 source files in the Pencil2D repository. But you only need to focus on a few of them to start with, the most important classes which form the backbone of Pencil2D. They are `MainWindow2`, `Editor`, `Object` and `ScribbleArea`. Once you get the idea of how these classes work, you will be able to pick up Pencil2D codebase very quickly.
19 
20 ## The entry point
21 
22 Let's start with `app/src/main.cpp`, the entry point of Pencil2D (and most of C++ programs as well). Skip the command-line handling code at the moment and go to the line `MainWindow2 mainWindow;` (it's at line 220 at the time of writing). The Pencil2D application starts here.
23 
24 ## MainWindow
25 
26 **MainWindow2**, as the name said, is the main window you will see when launching Pencil2D. Have a look at the constructor in `mainwindow2.cpp`, you will find the initialization process of Pencil2D. In the constructor, it creates an `Object` (An animation project which holds layers & frames), `Editor` (the central part of the animation engine), `ScribbleArea` (where you draw things) and all subpanels including tools, colour wheel, timeline etc.
27 
28 MainWindow acts as a bridge between the GUI widgets and the core engine.
29 
30 ### Menus
31 
32 It's good to pick a menu action, for example: **Open a Project** to start your journey in Pencil2D codebase.
33 
34 Firstly, go to `MainWindow2::createMenus()`, where all the menu actions are connected.
35 
36 Let's take the following line as an exmaple:
37 ```cpp
38 connect(ui->actionOpen, &QAction::triggered, this, &MainWindow2::openDocument);
39 ```
40 it tells you one thing: when a user hits **Open** from the File menu, it calls `MainWindow2::openDocument()`.
41 
42 Reference: [Qt signal & slot](http://doc.qt.io/qt-5/signalsandslots.html)
43 
44 And then you are able to figure out what actually happens when opening a file by seeing the function `MainWindow2::openDocument()`. (Tip: jump to the function by Ctrl + clicking the function name in QtCreator)
45 
46 Similarly, you can do this with all other menu actions. It will give you a good start point to learn the codebase.

48 ### Subpanels
49 
50 MainWindow holds all the subpanels (which are called **Widgets** in Qt's way). `MainWindow2::createDockWidgets()` creats and initialises subpanels. All subpanels have straight forward C++ class names like `Timeline` or `ToolBox`. Simply go to the corresponding cpp file for further details.

54 ## Object
55 
56 **Object** keeps your animation assets. Go to `object.h`, you can see that a `Object` object holds a list of Layers (`QList<Layer*> mLayers;`). And a layer holds keyframes. Opening a project is the process of converting a pclx file into a `Object` structure. Similarly, saving a project is a process of converting the Object structure back into a pclx file.
57 
58 Currently, there are 4 types of Layer: Bitmap, Vector, Sound and Camera layer. And 4 types of KeyFrames: BitmapImage, VectorImage, Soundclip and Camera. When a user adds a new bitmap key, it literally creates a BitmapImage, and then insert it into a Bitmap Layer.
59 
60 ## ActionCommands
61 
62 Every method in action commands is bound to an action in the menu/top bar. Use this method to trigger a behaviour or widget. As long the behaviour is directly bound to the action, it should be implemented here.
63 
64 **Example1**: You implement a new widget and bind the action to ActionCommands::foo()
65 
66 Foo opens a widget that does x instantly. This is fine.
67 Another outcome: Foo adds a new keyframe, this is also fine.
68 
69 **Example2**: You implement a new widget and bind the action to ActionsCommands::foo(), You trigger some behaviour via a button or other UI related command, now foo() calls ActionCommands::bah().
70 
71 This is wrong since bah() is not directly triggered by a command. bah() should instead be implement somewhere else where appropriate.

73 ## Editor
74 
75 The class that keeps the bridge between the UI/Widget related code and the core engine, as well as where all managers are handled.
76 
77 As of writing this the editor holds a property for
78 + ColorManager
79 + ToolManager
80 + LayerManager
81 + PlaybackManager
82 + Viewmanager
83 + PreferenceManager
84 + SoundManager

86 ## Manager classes
87 
88 All manager classes are required to derive from BaseManager, a class which contains two core members and some virtual methods.
89 + Editor
90 + Object
91 
92 The manager classes are to be independent of each other and should be treated as such. Their objective is to decouple logic from Editor and make it easier to organise and write tests.
93 
94 ## ScribbleArea
95 
96 The ScribbleArea is currently a collection point where preparations are made before content is being applied to the canvas. All widgets that interacts with the canvas in one with or another, goes through this class.
97 
98 Some things that are handled here:
99 - Event management (ie. mouse and tablet events)
100 - Selection related modifications and transformations
101 - Canvas cursor drawing
102 - Bitmap buffer painting
103 - Stroke dab making ie. all "drawFoo related methods"
